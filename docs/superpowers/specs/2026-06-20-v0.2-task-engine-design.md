# cms-tg-ingest v0.2 任务引擎设计

## 目标

v0.2 把当前可运行的单文件外挂升级为“可追踪、可重试、可修复”的任务引擎。用户继续通过 Telegram 发送 115 分享链接，系统仍保持单 Docker 容器、SQLite、CMS 主导整理的产品边界，但每条链接会拥有明确状态、阶段日志、失败原因和恢复入口。

## 成功标准

1. 每条 115 链接都有明确阶段：收到链接、提交 CMS、CMS 整理、自有分享、分享同步、STRM 生成、移动、Emby 确认、115 源清理。
2. 任一阶段失败时，任务记录包含失败阶段、错误摘要、可执行的下一步建议。
3. 支持从失败阶段重试，而不是只能整条链路重跑。
4. Telegram 保留轻量入口：任务详情、重试、质量巡检、打开 Web 管理页。
5. Web 管理页提供任务列表、任务详情、配置检查、规则查看和手动重试。
6. 代码从 `bridge.py` 单文件逐步拆分为模块，但不改变现有核心行为。
7. SQLite 继续作为唯一数据库，不引入外部数据库或复杂队列服务。
8. v0.1 的主要测试继续通过，现有环境可以平滑升级。

## 非目标

- 不重写 CMS 的整理、识别、重命名逻辑。
- 不替代 CMS 的分享同步能力。
- 不做多用户权限系统；继续以单允许 Telegram ID 为主。
- 不引入 React/Vue 等前端框架；Web 管理页使用标准库 HTTP + 简单 HTML。
- 不在 v0.2 做大规模数据库迁移或外部服务依赖。
- 不改变“Emby 确认后才清理 115 转存源，且不取消自有分享”的安全边界。

## 架构概览

v0.2 保持单容器运行，但拆出清晰边界：

```text
Telegram Bot  ─┐
Web Admin     ├─> Task Engine -> CMS / 115 / Emby / Filesystem
Repair Loop   ┘        │
                       └─> SQLite task store
```

任务引擎是核心：所有入口只提交“动作请求”，真正流程由状态机驱动。这样 Telegram、Web、后台修复循环都能复用同一套阶段判断和重试逻辑。

## 模块拆分

第一阶段只拆最小必要集合，避免一次性大重构。

- `app/config.py`：环境变量解析、默认值、配置校验。
- `app/store.py`：SQLite schema、任务读写、阶段事件、幂等查询。
- `app/models.py`：任务阶段、动作、结果、错误分类的数据结构。
- `app/clients/telegram.py`：Telegram API 和按钮回调。
- `app/clients/cms.py`：CMS 登录、任务提交、分享同步、自动整理。
- `app/clients/p115.py`：115 搜索、创建分享、删除转存源。
- `app/clients/emby.py`：Emby 刷新和入库确认。
- `app/workflow.py`：任务状态机和阶段调度。
- `app/move.py`：STRM 路径选择、移动、合并、安全路径检查。
- `app/quality.py`：任务质量巡检、STRM 类型检查、Emby 匹配检查。
- `app/web.py`：轻量 Web 管理页。
- `bridge.py`：保留为入口兼容层，逐步变薄。

## 任务状态机

任务阶段使用固定枚举，避免自由字符串继续膨胀：

- `received`：收到链接并完成去重。
- `cms_submitted`：已提交 CMS 转存任务。
- `organized`：找到 CMS 整理后的 115 文件夹。
- `own_share_created`：已创建自有 115 永久分享。
- `share_sync_submitted`：已调用 CMS 分享同步。
- `strm_ready`：在分享 STRM 根目录找到完整 STRM 文件夹。
- `moved`：STRM 文件夹已移动或合并到媒体库。
- `emby_confirmed`：Emby 已确认入库。
- `cleaned`：115 转存源已删除，自有分享保留。
- `needs_action`：需要人工选择或确认。
- `failed`：阶段失败，等待重试或人工处理。

每个阶段写入一条事件：阶段、开始时间、结束时间、状态、摘要、错误类型、错误详情。任务表只保留当前阶段和关键索引字段，完整过程从事件表查看。

## 重试和修复

v0.2 的重试原则：从最近失败或未完成阶段继续，不默认从头提交。

示例：

- `share_sync_submitted` 后找不到 STRM：重试只重新检查 STRM 或重新提交分享同步。
- `moved` 后 Emby 未确认：重试只重新刷新/查询 Emby。
- `emby_confirmed` 后清理失败：重试只删除 115 转存源。
- 分类不确定：进入 `needs_action`，等待 Telegram 或 Web 操作。

后台 repair loop 继续存在，但改为读取状态机阶段，而不是依赖多列状态猜测。

## Telegram 交互

Telegram 仍是主要日常入口，但只承载高频操作：

- 裸链接提交。
- 完成通知。
- 失败通知，附带“查看详情”“重试当前阶段”“打开 Web”。
- `/status` 展示最近任务摘要。
- `/quality` 展示需要处理的问题。
- 分类不确定时给按钮选择。

Telegram 不展示完整日志，避免消息过长。完整排查进入 Web 管理页。

## Web 管理页

Web 管理页是轻量标准库页面，不引入前端构建链。

页面范围：

1. `/`：任务列表，显示标题、阶段、媒体库、更新时间、错误摘要。
2. `/task/<id>`：任务详情，显示时间线、路径、分享码、STRM URL 类型、Emby 命中、清理状态。
3. `/task/<id>/retry`：POST，从当前失败阶段重试。
4. `/health`：配置检查结果，包括 CMS、115 cookie、STRM 路径、Emby。
5. `/quality`：质量巡检问题列表。
6. `/rules`：展示分类映射和路径映射，v0.2 只读。

鉴权：v0.2 默认只监听容器内或局域网地址，通过部署者网络隔离。可选 `WEB_TOKEN`，启用后访问管理页需要 token。

## 数据模型

新增或调整为两层：

`tasks`：当前状态和索引字段。

- `id`
- `share_code`
- `receive_code`
- `url`
- `title`
- `tmdb_id`
- `category`
- `current_stage`
- `status`
- `error_type`
- `error_summary`
- `retry_count`
- `created_at`
- `updated_at`

`task_events`：阶段时间线。

- `id`
- `task_id`
- `stage`
- `status`
- `message`
- `error_type`
- `error_detail`
- `created_at`

现有 `submissions` 表先保留。迁移策略是兼容读取旧表，并把新任务写入新表；v0.2 不强制破坏性迁移。

## 质量巡检

质量巡检聚焦真实故障：

- STRM 内容是否是 `/s/<自有分享>_1212_...`，不是 `/d/` 直链。
- 目标路径是否存在。
- 目标路径是否有 `.strm`。
- Emby 是否能按 TMDB 或路径命中。
- `moved` 但 `emby_confirmed` 缺失的任务。
- `emby_confirmed` 但 `cleaned` 缺失的任务。
- 自有分享存在但 STRM 文件夹丢失的任务。

Web 和 Telegram `/quality` 都读取同一套 `quality.py` 结果。

## 配置

保留 `.env`，新增：

- `WEB_ENABLED=true`
- `WEB_HOST=0.0.0.0`
- `WEB_PORT=8787`
- `WEB_TOKEN=`
- `TASK_WORKER_INTERVAL_SECONDS=5`
- `TASK_MAX_RETRIES=3`

v0.2 不提供在线编辑 `.env`，只做展示和校验，避免 Web 改坏运行配置。

## 升级策略

1. 保留 `bridge.py` 入口，Docker 命令不变。
2. 启动时执行 SQLite additive migration。
3. 旧 `submissions` 数据不删除。
4. v0.2 可读取旧记录用于 `/status` 和质量巡检，但新任务写入 `tasks/task_events`。
5. 如果升级失败，回退镜像和数据库文件备份即可。

## 测试策略

自动化测试：

- 状态机阶段推进。
- 每个失败阶段的重试入口。
- 旧 `submissions` 兼容读取。
- Telegram 回调权限和按钮行为。
- Web 路由返回任务列表、详情、重试结果。
- `quality.py` 对直链 STRM、缺失 Emby、缺失清理的识别。
- 配置校验和 `doctor.py` 继续工作。

手工验证：

1. 本地测试全部通过。
2. Docker 镜像可构建。
3. 在 Unraid 测试一条小体量 115 链接。
4. 验证 TG 通知、Web 详情、Emby 入库、115 源清理。
5. 人为制造一个失败阶段，验证“重试当前阶段”。
