# HDHive 剧集订阅设计

日期：2026-07-24  
状态：已确认，待实现计划

## 目标

在现有单用户 Telegram/CMS 入库程序中增加 HDHive 剧集订阅。用户可以：

- 从 HDHive 搜索结果点击“订阅此剧”；
- 直接发送 `https://hdhive.com/tv/<slug>` 订阅，例如 `https://hdhive.com/tv/542a1c1fe6ac4a5aab152369079596b5`；
- 每天默认在 `Asia/Shanghai` 的 `01:30` 检查新增剧集；
- 对新增的有效 115 资源自动解锁，并复用现有 115 整理、分享 STRM、移动和 Emby 入库链路。

电影暂不支持订阅。

## 已确认规则

- 订阅对象是电视剧，不是电影。
- 默认网盘过滤为 115。
- 同一集多个资源的选择优先级固定为：
  1. 资源有效；
  2. 分辨率最高；
  3. 免费或积分成本最低。
- 每一集只选择一个资源，避免重复扣费和重复入库。
- 费用不超过 20 分或资源已属于当前账号时自动解锁。
- 费用超过 20 分或费用未知时进入待确认，不自动扣积分。
- 同一资源成功解锁或已提交入库后永久去重。
- 订阅任务顺序执行，不提高 115 或 HDHive 请求并发。

## 直接 HDHive URL

只接受 HTTPS/HTTP 的 `hdhive.com` 或 `www.hdhive.com` 电视剧路径：

```text
https://hdhive.com/tv/<slug>
```

程序提取并保存 `<slug>` 作为来源标识，然后读取该页面的服务端元数据，解析页面中的 `tmdb_id`、剧名和年份。页面解析成功后，后续资源查询使用现有的 CMS OAuth 代理和 `tv + tmdb_id` 接口，不调用未确认的 HDHive 私有“按页面 slug 查询资源”接口。

页面无法访问、不是电视剧、或缺少 TMDB ID 时拒绝创建订阅，并明确提示用户先确认链接有效。已创建的订阅会持久化 `source_url`、`hdhive_slug` 和解析出的 `tmdb_id`，后续检查不重复抓取页面。

## 架构

### 订阅存储

新增 `HdhiveSubscriptionStore`，使用现有 `TASK_DB_PATH` 指向的 SQLite 数据库，避免增加第二个数据库和部署卷。存储分为三类记录：

- 订阅：来源 URL/来源类型、TMDB ID、标题、状态、网盘过滤、检查时间、最近结果；
- 集资源状态：资源 slug、集数标识、已选资源、资源状态、关联 TaskStore 任务号；
- 执行记录：运行日期、开始/结束时间、发现数量、自动解锁数量、待确认数量、失败数量和错误摘要。

订阅唯一键为 `chat_id + source_type + source_value`。资源状态写入数据库并采用唯一约束，重启或重复点击不会重复解锁。

集数标识优先使用代理返回的季/集字段；没有结构化字段时，从资源标题解析 `SxxEyy`，仍无法解析时退回资源 slug。退回 slug 可以保证不误合并不同资源，但不会强行猜测集数。

### 检查调度

新增 `HdhiveSubscriptionScheduler`，结构与现有质量巡检调度一致：

- 使用本地时区计算每日计划时间，默认 `01:30`；
- 使用 SQLite 租约和本地日期标记，避免重启、重复线程或多实例重复执行；
- 逐条执行启用中的订阅；
- 每个订阅内部按资源查询、筛选、解锁、提交入库顺序处理；
- 单个订阅异常只记录该订阅，不阻塞其他订阅；
- 网络错误、授权过期和风控冷却沿用现有重试/冷却机制。

Web 可以修改执行时间和时区，也可以手动触发一次检查；Telegram 的“立即检查”只触发指定订阅。

### 资源选择与解锁

每次检查执行以下步骤：

1. 查询订阅对应的 TV 资源；
2. 过滤非 115 资源；
3. 排除明确无效的资源；
4. 按集数标识分组；
5. 每组按“有效性、分辨率、费用”排序，选出一个候选；
6. 已成功解锁、已提交或已有 TaskStore 任务的资源跳过；
7. 费用不超过阈值或已拥有时调用解锁；
8. 高费用/未知费用写入待确认状态并通知用户；
9. 成功得到 115 分享链接后调用现有 intake helper，后续完全由现有 TaskStore/CMS 工作流执行。

订阅模块不直接操作 115 文件，不直接生成 STRM，也不复制 CMS 整理逻辑。解锁结果只通过现有入库入口进入主任务队列。

### Telegram

- HDHive 搜索候选和资源页面增加“订阅此剧”；
- 直接发送 HDHive TV URL 自动创建订阅，不触发解锁；
- 菜单增加“HDHive 订阅”，显示订阅列表和下次检查时间；
- 每条订阅提供暂停、恢复、删除、立即检查；
- 高费用资源提供“确认解锁”；
- 通知只发送新增资源、待确认、失败和最终入库提交结果，避免每天重复刷屏。

### Web

新增 HDHive 管理页面，包含：

- CMS OAuth/HDHive 账号状态；
- 订阅来源 URL、剧名、TMDB ID、启用状态；
- 最近检查、下次检查、发现/解锁/待确认/失败统计；
- 暂停、恢复、删除、立即检查；
- 全局检查时间及时区配置；
- 待确认高费用资源及确认操作。

页面沿用当前 Web 管理端的鉴权和简单 HTML 结构，不引入前端框架或独立 API 服务。

## 状态与错误处理

订阅状态：`active`、`paused`、`error`、`deleted`。资源状态：`discovered`、`pending_confirmation`、`unlocking`、`enqueued`、`failed`。

- 页面解析失败：不创建新订阅；已有订阅保留上次有效 TMDB 信息并记录错误；
- HDHive 查询失败：不标记资源已处理，下次继续检查；
- 资源无效：记录并跳过，不解锁；
- 解锁部分失败：成功项进入入库，失败项保留错误并允许手动重试；
- CMS/115 风控冷却：暂停当前执行，保留订阅状态，不重复扫描；
- 入库提交失败：保留解锁结果和资源状态，允许从订阅页重试，不再次解锁同一资源；
- 删除订阅只停止后续检查，不删除已经入库的 TaskStore 任务或媒体文件。

所有日志和 Telegram 消息不得输出 OAuth token、refresh token 或完整解锁链接；已有任务库中保存链接的行为保持不变。

## 配置

HDHive 功能关闭时不启动订阅调度。启用 HDHive 后，订阅调度默认启用并使用：

```env
HDHIVE_SUBSCRIPTION_AUTO_ENABLED=true
HDHIVE_SUBSCRIPTION_TIME=01:30
HDHIVE_SUBSCRIPTION_TIMEZONE=Asia/Shanghai
```

Web 保存的运行时设置优先于环境变量；恢复默认时回到环境变量值。单条订阅的暂停状态优先于全局开关。

## 验证

自动化测试覆盖：

- HDHive URL 域名、路径和 slug 校验；
- 页面元数据 TMDB ID 解析和无效页面拒绝；
- 订阅唯一性、重启后去重和资源状态迁移；
- `01:30` 与 `Asia/Shanghai` 的计划计算及执行租约；
- 有效性/分辨率/费用选择优先级；
- 20 分边界、未知费用确认和部分解锁失败；
- 成功的 115 链接只进入一次现有入库入口；
- Telegram 按钮和直接 URL 创建订阅；
- Web 暂停/恢复/删除/立即检查/时间设置；
- 单个订阅失败不影响其他订阅。

真实环境验证先执行账号状态、页面解析、资源查询和订阅检查预览；只有用户确认高费用资源后才允许真实解锁。
