# 后端重构设计：TaskRunner 自分享工作流收口

## 背景

`cms-tg-ingest` 已经能完成 115 分享链接到 Emby 入库的一条龙流程，但后端逻辑逐步堆叠在 `bridge.py` 中，职责混杂：Telegram 入口、CMS 调用、115 调用、Emby 确认、STRM 文件操作、分类识别、任务状态推进和旧轮询兼容都集中在一个大文件里。

近期问题集中在三类：

1. 新 TaskRunner 流程和旧 SubmissionStore 轮询路径并存，容易出现旧状态干扰或用户“发了没反应”。
2. CMS 普通同步产生的直链 STRM 可能混入媒体库，和目标“只使用自己的 115 分享链接 STRM”冲突。
3. 代码边界不清，导致修复分类、STRM、清理、Emby 确认等问题时容易牵一发动全身。

本重构优先解决后端稳定性和可维护性；Web 只保持基本状态展示和按钮能力，不作为重构重点。

## 目标

- 保持现有外部行为不变：Telegram 裸链接仍自动触发完整流程。
- 拆分 `bridge.py` 的后端职责，让每个模块可独立测试和维护。
- 新链接只由 TaskRunner 推进真实执行状态，SubmissionStore 仅做历史兼容。
- 明确禁止直链 STRM 被当作最终入库结果。
- 减少 115 请求范围和频率，不新增任何广泛扫描行为。
- 保留现有测试覆盖，并在关键安全点补充回归测试。

## 非目标

- 不重写 Web 管理页 UI。
- 不更换数据库技术，继续使用 SQLite。
- 不引入复杂队列、中间件或异步框架。
- 不改变 CMS、115、Emby 的对外配置语义。
- 不做高并发优化；当前工具以单用户、低并发、稳定执行为核心。

## 总体方案

采用“小步组合”的重构路线：

1. 先按职责拆分后端模块，尽量保持行为不变。
2. 再把新自分享任务收口到 TaskRunner 状态机。
3. 最后建立统一 STRM 安全闸门，确保最终入库只能使用自己的分享链接 STRM。

这种方式避免一次性大重写，降低正在使用中的 Docker 程序被改坏的风险。

## 模块边界

### `bridge.py`

保留为薄入口。职责：

- 加载配置。
- 初始化客户端和 store。
- 启动 Telegram bot、TaskRunner、Web server。
- 保留必要的兼容导入，降低测试和旧入口迁移风险。

`bridge.py` 不再承载核心业务流程。

### `app/config.py`

职责：

- `Config` 数据结构。
- 环境变量解析。
- 布尔、浮点、列表、路径映射解析。
- 默认媒体库路径和分类常量。

### `app/clients/cms.py`

职责：

- CMS 登录和授权请求。
- 提交分享转存或分享同步。
- 触发 CMS 自动整理。
- 查询 CMS 配置和任务状态。

原则：CMS 普通同步和 CMS 分享同步必须显式区分，TaskRunner 自分享流程只允许调用分享同步。

### `app/clients/p115.py`

职责：

- 115 cookie 加载。
- 请求限速。
- 接收外部分享。
- 搜索已整理文件夹。
- 创建自己的永久分享。
- 删除 115 转存源。

原则：默认只做目标搜索，不做广泛递归扫描；如未来需要扫描，必须由显式配置打开。

### `app/clients/emby.py`

职责：

- Emby API 请求。
- 根据路径刷新对应媒体库。
- 根据 TMDB 查询媒体条目。
- 返回入库媒体库名称。

### `app/media/strm.py`

职责：

- 查找 STRM 源目录。
- 判断 STRM 是直链还是分享链。
- 校验自分享 STRM。
- 合并 STRM 到目标媒体库。
- 删除旧直链 STRM。
- 本地质量巡检所需的文件检查。

这是防止直链 STRM 入库的核心模块。

### `app/media/classify.py`

职责：

- 从 CMS 整理后的文件夹、父目录、已有媒体库目录推断分类。
- 使用 TMDB 作为辅助信息。
- 判断是否需要人工确认分类。

原则：CMS 分类优先，TMDB 只辅助，不压过 CMS 的整理结果。

### `app/workflows/self_share.py`

职责：

- 实现自分享工作流阶段逻辑。
- 每个阶段只调用客户端、分类模块和 STRM 模块。
- 不直接操作 Telegram 消息格式。

阶段固定为：

1. `received`：接收 115 外部分享到待整理 CID。
2. `organizing`：只触发 CMS 自动整理，不触发 CMS 普通同步。
3. `recognizing`：以 CMS 整理后的目录和分类为主识别分类。
4. `own_share_created`：对整理后的 115 文件夹创建自己的永久分享。
5. `share_sync_submitted`：调用 CMS 分享同步，使用自己的分享链接生成 STRM。
6. `strm_ready`：等待 `/strm/share` 下出现分享 STRM。
7. `moved`：移动或合并到 `/strm/转存/...` 目标媒体库。
8. `emby_confirmed`：刷新并确认入库。
9. `cleaned`：删除 115 转存源，但不取消自己的分享。

### `app/telegram/bot.py`

职责：

- Telegram 长轮询。
- 命令和按钮解析。
- 消息格式化。
- 把裸链接转交给任务创建逻辑。

Telegram 层不直接执行 CMS、115、STRM 逻辑。

## TaskRunner 收口规则

新链接的真实执行状态以 TaskStore 为准。规则如下：

- `TASK_ENGINE_ENABLED=true` 时，新 self-share 链接只创建 TaskStore 任务。
- TaskRunner 是唯一推进真实阶段的执行器。
- SubmissionStore 保留用于历史展示、兼容旧记录、关联旧字段。
- `/status`、`/history`、Web 详情优先读 TaskStore，必要时回退到 SubmissionStore。
- “重试当前阶段”“恢复 STRM”“从头重跑”等操作必须落到 TaskStore 阶段，不直接调用旧轮询流程。

成功标准：

- 发新链接后只看到一条 TaskStore 任务推进。
- 不自动触发 CMS 普通同步生成最终 STRM。
- 如果 CMS 自己生成直链 STRM，只能作为分类参考，不能被移动入库。
- 任务卡住时能明确显示卡在哪个阶段。

## STRM 安全闸门

自分享工作流移动入库前必须调用统一校验。

校验规则：

- STRM 源目录必须位于 `SELF_SHARE_STRM_ROOT`。
- 每个 `.strm` 内容必须包含 `/s/<own_share_code>_<receive_code>_`。
- 任意 `.strm` 内容包含 `/d/` 时直接阻断移动。
- 文件夹 TMDB 和任务 TMDB 同时存在且不一致时直接阻断移动。
- 目标媒体库路径必须位于允许的媒体库根目录下。

合并规则：

- 目标目录不存在时，移动整个分享 STRM 目录。
- 目标目录存在时，先删除目标目录内旧直链 `.strm`，再复制分享 STRM。
- 非 `.strm` 文件尽量保留。
- 合并完成后删除临时分享 STRM 源目录。

等待规则：

- 如果分享 STRM 尚未生成，任务停留在 `strm_ready`。
- 不允许降级使用 CMS 普通同步产生的直链 STRM。
- 错误信息明确显示“等待分享同步 STRM”或具体校验失败原因。

## 分类原则

分类优先级：

1. 人工已确认分类。
2. CMS 整理后的 115 父目录 CID 映射。
3. 已存在媒体库目录匹配。
4. CMS 整理后的文件夹名和 TMDB 信息辅助判断。
5. 人工确认。

TMDB 用于补充媒体类型和标题，不应在 CMS 已经明确分类时覆盖 CMS 结果。

OpenAI 分类不是默认路径，后续只作为人工辅助建议，不参与自动最终分类。

## 数据和兼容

- 保留 `submissions.db`，避免破坏历史记录和现有命令输出。
- 新任务主状态写入 `tasks.db`。
- SubmissionStore 中和新任务有关的字段可继续同步写入，用于兼容旧展示和迁移期排障。
- 不做数据库大迁移；如需新增字段，必须有向后兼容默认值。

## 测试策略

重构必须先写或保留以下测试：

- 模块拆分后现有测试全部通过。
- TaskRunner 新链接不会调用旧轮询真实流程。
- 自分享流程不会调用 CMS 普通同步作为最终 STRM 来源。
- STRM 安全闸门拒绝 `/d/` 直链。
- STRM 安全闸门接受 `/s/<own_share_code>_<receive_code>_` 分享链。
- 目标目录已存在时，只删除直链 STRM 并合并分享 STRM。
- CMS 直链目录只可用于分类参考，不能作为最终移动源。
- 115 客户端搜索保持目标搜索，不引入广泛递归扫描。

本地验证命令：

```sh
python3 -m py_compile bridge.py doctor.py app/*.py
python3 -W error::ResourceWarning -m unittest discover -s tests -v
```

## 实施顺序

1. 提取纯工具和配置模块，保持导入兼容。
2. 提取 CMS、115、Emby 客户端。
3. 提取 STRM 文件操作模块，并补安全闸门测试。
4. 提取分类模块。
5. 提取 self-share workflow。
6. 收口 TaskRunner 新链接路径，限制旧轮询真实执行。
7. 清理 `bridge.py`，让它只保留入口和兼容胶水。
8. 更新 README 中的架构和运维说明。

每一步都应小提交，并保持测试通过。

## 风险和缓解

- 风险：模块拆分时破坏旧测试导入。
  - 缓解：`bridge.py` 迁移期保留兼容 re-export。
- 风险：TaskRunner 收口后旧记录展示异常。
  - 缓解：查询层继续 fallback 到 SubmissionStore。
- 风险：过早删除 115 源导致无法修复。
  - 缓解：只在自己的永久分享创建成功后删除源文件，不取消分享。
- 风险：为了分类做过多 115 请求。
  - 缓解：默认目标搜索；广泛扫描必须显式配置且不在本次重构新增。
- 风险：Web 页面短期能力不完整。
  - 缓解：Web 不是重点，只保留状态和基础按钮。

## 完成定义

- `bridge.py` 明显变薄，核心后端职责拆入模块。
- 新链接执行路径由 TaskRunner 主导。
- 分享 STRM 安全闸门统一生效。
- 本地测试全通过。
- README 说明新的后端结构和安全规则。
- 不引入新的 115 高频扫描行为。
