# v0.2 Alpha.2 TaskStore Bridge Design

## Goal

Wire the real Telegram/CMS/self-share workflow into the v0.2 `TaskStore` as a sidecar task timeline, without replacing the existing `SubmissionStore` runtime path.

## Current State

v0.2 alpha.1 added:

- `app.models`: task stages, statuses, retry decisions, and snapshots.
- `app.task_store`: SQLite `tasks` and `task_events` tables.
- `app.web`: task list, task detail, and manual retry event endpoint.
- `bridge.py` config and optional web server startup.

The production workflow still records real progress in `SubmissionStore`. `TaskStore` exists, but normal Telegram link ingestion does not yet populate it. As a result, the Web task admin is structurally ready but not useful for live production work.

## Scope

Alpha.2 adds sidecar synchronization from the existing workflow to `TaskStore`.

In scope:

- Create a focused mapping module that converts existing submission rows and workflow events into `TaskStage` / `TaskStatus` events.
- Add optional `task_store` parameters to `handle_update()` and `start_status_poll()`.
- Record task timeline events for new links and key asynchronous workflow transitions.
- Keep `SubmissionStore` as the source of truth for actual work execution.
- Keep all existing Telegram/CMS/self-share behavior compatible when no `TaskStore` is supplied.
- Add tests covering both direct CMS submission and self-share/STRM/Emby progress mapping.

Out of scope:

- Replacing `SubmissionStore` with `TaskStore`.
- A dedicated task worker that drives workflow stages.
- Automatic destructive or semi-destructive retry from Web admin.
- Migration/backfill of old `SubmissionStore` rows into `TaskStore`.
- Changes to 115, CMS, or Emby API behavior.

## Architecture

Add `app/task_bridge.py` as a small sidecar adapter. It is responsible for idempotently creating tasks and recording events in `TaskStore` from the current workflow. `bridge.py` remains the runtime coordinator.

`TaskStore` is a projection of workflow state, not the execution authority. If sidecar recording fails, the existing workflow must continue and only log the failure. This avoids breaking the user's already-working ingestion path.

## Task Mapping

The bridge module uses the following mapping:

| Existing workflow signal | Task stage | Task status | Notes |
| --- | --- | --- | --- |
| link parsed / accepted | `received` | `pending` | Create task keyed by share code + receive code. |
| `cms.add_share_down()` succeeded | `cms_submitted` | `running` | Include CMS task id/title when available. |
| CMS status/detail update | `organized` | `running` or `succeeded` | Use current title/category metadata when known. |
| category prompt needed | `needs_action` | `needs_action` | Error summary explains user input is needed. |
| self-share file/folder found and own share created | `own_share_created` | `succeeded` | Include own share URL/code if available in message/detail. |
| CMS share sync submitted | `share_sync_submitted` | `running` | Do not mark STRM ready until move source exists or move completes. |
| STRM move source found / move attempted | `strm_ready` | `running` or `failed` | Missing source maps to failed or remains pending depending retryability. |
| STRM folder moved/merged | `moved` | `succeeded` | Include destination path/category. |
| Emby match confirmed | `emby_confirmed` | `succeeded` | Include Emby title/library/path. |
| Emby timeout/disabled | `emby_confirmed` | `failed` or `needs_action` | Timeout is failed; disabled is needs_action with summary. |
| cleanup deleted source | `cleaned` | `succeeded` | Only after `cleanup_status=deleted`. |
| operation exception | current stage or `failed` | `failed` | Store classified error summary. |

## Public Interfaces

`app/task_bridge.py` exposes:

- `ensure_task_for_link(task_store, share_code, receive_code, url) -> TaskSnapshot`
- `record_submission_event(task_store, row, stage, status, message, **metadata) -> TaskSnapshot | None`
- `record_failure(task_store, row_or_key, stage, error_summary, error_type="", error_detail="") -> TaskSnapshot | None`
- `sync_task_from_submission(task_store, row, message="同步现有任务状态") -> TaskSnapshot | None`

These functions are defensive:

- Accept `None` task store and return `None`.
- Catch expected missing-row/unknown-field cases with safe defaults.
- Do not swallow database errors silently inside the module; callers in `bridge.py` wrap sync calls to protect production flow.

## bridge.py Integration Points

`handle_update()` gains an optional `task_store: TaskStore | None = None` parameter.

For each link:

1. After parsing the share link, call `ensure_task_for_link()` and record `received` if this is a new task.
2. If an existing active submission is found, record a lightweight `received/pending` event or update timestamp, then skip duplicate submission as today.
3. After `cms.add_share_down()` succeeds and `SubmissionStore.upsert_submission()` returns a row, record `cms_submitted/running`.
4. On submission exception, best-effort record `failed/failed` if the link can be parsed.
5. Pass `task_store` into `start_status_poll()`.

`start_status_poll()` gains optional `task_store: TaskStore | None = None`.

Inside the worker:

- When CMS task id is discovered later, record `cms_submitted/running`.
- On each CMS detail status change, record `organized/running` and include title.
- When category decision requires a Telegram prompt, record `needs_action/needs_action`.
- After self-share `prepare()` returns a row, inspect row fields:
  - `own_share_code` present => `own_share_created/succeeded`.
  - `share_sync_status=submitted` or `restore_submitted` => `share_sync_submitted/running`.
- Before/after move planning:
  - Retryable missing/unstable STRM source does not create a failure event every poll; it records at most a pending/running `strm_ready` event for visibility.
  - Non-retryable move failure records `strm_ready/failed` or `moved/failed` with reason.
  - Successful move records `moved/succeeded`.
- On Emby confirmation, record `emby_confirmed/succeeded` before returning.
- On Emby timeout, record `emby_confirmed/failed`.
- On Emby disabled, record `emby_confirmed/needs_action`.
- After cleanup deletes the source, record `cleaned/succeeded`.

`run_forever()` already creates `task_store`. It should pass it into `handle_update()`.

## Event Idempotency

Alpha.2 can tolerate duplicate event rows, but it must avoid excessive spam during polling.

Rules:

- `TaskStore.upsert_task()` remains idempotent by `(share_code, receive_code)`.
- Mapping helpers can call `record_event()` repeatedly for meaningful stage/status changes.
- Poll-loop visibility events should only be written when a relevant row field changes or a major stage is first reached.
- A small helper may compare the current `TaskSnapshot` before writing to skip exact duplicates for the same stage/status/error/message.

## Failure Behavior

TaskStore synchronization must not break ingestion.

- Every `bridge.py` call into `task_bridge` should use a best-effort wrapper that logs `LOG.debug(..., exc_info=True)` on sync errors and continues.
- Existing Telegram user-facing messages remain unchanged in alpha.2 unless a test proves a message must mention task ids.
- Database errors in `TaskStore` do not roll back or block `SubmissionStore` updates.

## Testing Strategy

New tests:

- `tests/test_task_bridge.py`
  - Ensures a task is created from a parsed link.
  - Ensures submission rows map to expected task stages.
  - Ensures missing optional fields do not crash mapping.
  - Ensures duplicate sync does not create noisy duplicate events when stage/status/message are unchanged.

- Extend `tests/test_bridge_v02_integration.py`
  - `handle_update()` with a fake CMS and fake Telegram records `received` and `cms_submitted` in `TaskStore`.
  - Duplicate submitted links do not submit to CMS again but keep TaskStore consistent.

- Extend existing self-share/status tests or add `tests/test_taskstore_workflow_events.py`
  - Polling a CMS status update records `organized`.
  - A self-share prepare row with `own_share_code` and `share_sync_status=submitted` records `own_share_created` and `share_sync_submitted`.
  - Move success records `moved`.
  - Emby confirmation records `emby_confirmed`.

Regression tests:

- Existing tests still pass when no `task_store` is provided.
- Full test suite stays warning-clean under `-W error::ResourceWarning`.
- Docker image import check remains valid.

## Release Notes

README/CHANGELOG should describe alpha.2 as:

- Web task admin now reflects real Telegram/CMS workflow progress for new links.
- TaskStore remains a sidecar timeline; current production execution still uses the existing stable workflow.
- Web retry is still non-destructive and records retry intent only.

## Risks and Mitigations

- Risk: duplicate events during status polling.
  - Mitigation: skip exact duplicate stage/status/message writes.

- Risk: TaskStore sync error breaks production ingestion.
  - Mitigation: all bridge integration points are best-effort wrappers.

- Risk: stage mapping overstates progress.
  - Mitigation: prefer `running` until the existing row has explicit completion fields such as `move_status=moved`, `emby_status=confirmed`, or `cleanup_status=deleted`.

- Risk: `bridge.py` gets even larger.
  - Mitigation: keep mapping logic in `app/task_bridge.py`; `bridge.py` only calls small helper functions.

## Success Criteria

- Sending a new 115 link through `handle_update()` creates a corresponding TaskStore task and events.
- The Web task list can show real tasks from the workflow, not only manually created test data.
- Existing `SubmissionStore` behavior remains compatible without `TaskStore`.
- All tests pass.
- Docker build and image import pass.
- Secret scan passes with only the expected diagnostics redaction regex match.
