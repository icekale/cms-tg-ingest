# 质量巡检全自动修复设计

## 背景

当前质量巡检只读取 TaskStore 和本地 STRM 文件，Web 的“修复全部”可以处理缺失目录、缺失 STRM、直链 STRM 和异常分享 STRM，但需要人工打开页面并点击修复。115 分享失效、STRM 与媒体库不一致、Emby 未确认等情况仍需要人工判断。

目标是在不引入第二个容器的前提下，让现有程序每天自动巡检并修复确定性问题，降低误分类、误移动、误删除和重复执行的概率。

## 目标

- 默认每天 `02:50`（`Asia/Shanghai`）执行一次质量巡检。
- 执行时间、时区、启用状态和每次处理上限可配置。
- 程序重启、任务并发和重复触发不会让同一天重复执行巡检。
- 可确定修复的问题自动处理，无法确定的问题只记录并通知 Telegram。
- 新 STRM、目标目录和 Emby 验证成功前，不删除旧 STRM 或 115 源文件。
- Web 显示自动巡检状态、最近结果、下次执行时间和任务级修复记录。
- 正常修复不发送 TG；失败、风控、疑似错配和人工介入才通知。

## 非目标

- 不扫描整个 115 网盘，不通过质量巡检触发 CMS 普通同步。
- 不自动覆盖 CMS 的分类和 TMDB 识别结果。
- 不在无法确认任务归属时猜测媒体库目录。
- 不新增独立巡检容器或外部数据库。

## 运行配置

环境变量提供默认值：

```env
QUALITY_AUTO_ENABLED=true
QUALITY_AUTO_TIME=02:50
QUALITY_AUTO_TIMEZONE=Asia/Shanghai
QUALITY_AUTO_MAX_TASKS=50
QUALITY_AUTO_115_CHECK_LIMIT=3
```

配置规则：

- `QUALITY_AUTO_TIME` 只接受 `HH:MM`，小时 0-23、分钟 0-59。
- `QUALITY_AUTO_TIMEZONE` 使用 Python `zoneinfo` 支持的 IANA 时区；无效时启动诊断失败，不静默回退。
- Web 设置保存在 TaskStore 的运行状态中，优先级高于环境默认值；页面提供“恢复环境默认”操作。
- 修改设置只影响下一次巡检，不中断正在执行的巡检。
- `QUALITY_AUTO_115_CHECK_LIMIT` 限制单次运行确认失效分享的任务数量，复用现有最小请求间隔和风控冷却。

## 架构

新增 `app/quality_automation.py`，提供三个边界清晰的入口：

- `run_if_due(now=None)`：判断当前本地日期和时间，使用 TaskStore 原子状态确保每天只认领一次。
- `run_once(run_id, limit=None)`：执行一次扫描、分类、修复和汇总。
- `run_now()`：供 Web 手动触发，后台执行，避免 HTTP 请求等待外部服务。

bridge 启动时只启动一个质量调度线程。线程每 30-60 秒检查一次本地时间，不调用 115；只有到执行时间并成功认领当天运行权时，才调用 `run_once`。任务引擎和质量调度器共用 TaskStore 任务锁，已有任务正在使用相同资源时，巡检只记录跳过原因。

TaskStore 复用 `runtime_state` 保存全局运行信息：

- 当前运行 ID、运行日期、运行状态和开始/结束时间。
- 最近一次结果摘要和下次执行时间。
- 最近一次错误及通知状态。

每个任务的具体修复通过既有 TaskStore 事件记录，事件消息统一带 `自动巡检` 和运行 ID。这样不需要增加第二套任务表，任务详情页可以直接看到修复过程。

## 自动处理流程

### 1. 本地扫描

复用 `scan_task_quality`，并增加自动修复资格判断。只处理有完整 TaskStore 记录、来源类型明确、目标路径在允许的 STRM/媒体库根目录内的任务。

问题分类：

- `missing_dest`：目标目录不存在。
- `missing_strm`：目标目录存在但没有 STRM。
- `direct_strm`：发现 `/d/` 直链 STRM。
- `unexpected_strm`：STRM 不是任务自己的分享标记。
- `invalid_share`：115 明确返回分享不存在、失效或取消。
- `emby_mismatch`：Emby TMDB、标题或路径与任务不一致。

### 2. 确定性修复

- 缺失目录或 STRM：复用现有恢复 STRM 流程，恢复 CMS 标准目录名，移动到原媒体库，然后刷新并确认 Emby。
- 直链或异常分享 STRM：复用现有从头重跑流程；仅在任务未运行、未被占用且重试次数未超限时入队。
- 失效自有分享：只检查当前任务已保存的自有分享，不扫描全盘。115 明确确认失效后，创建中性别名的新永久分享，调用分享同步生成新 STRM，验证新标记、播放端点、目标目录和 Emby 后，才清理旧 STRM 和已确认的 115 转存源。
- Emby 未确认：允许再次刷新和查询，但不把“刷新请求成功”当作“入库成功”；超时进入待处理并通知。

已有的失效分享探测循环与自动巡检只能保留一个调度来源。启用每日巡检后，旧的独立失效分享探测不再并行运行，避免重复 115 请求。

### 3. 跳过和人工处理

以下情况不自动修改文件、不删除内容：

- 任务正在运行、已被其他 worker 占用或锁信息异常。
- 没有可靠的自有分享码、任务来源或 CMS 目录元数据。
- 分类、TMDB、目标媒体库存在多个候选。
- 115 处于风控冷却、返回未知错误或分享状态无法确认。
- 新 STRM 不包含当前自有分享标记，播放探测失败，或 Emby 命中多个不一致项目。
- 达到每日任务上限或任务最大重试次数。

跳过项记录明确原因；失败项保留当前文件和分享，并进入 Web/TG 的“需关注”列表。

## 删除保护

删除必须满足全部条件：

1. 新自有分享已创建且状态可用。
2. 新 STRM 使用当前自有分享链接，不是 `/d/` 直链或旧分享。
3. 新 STRM 已移动到允许的媒体库根目录。
4. Emby 已通过 TMDB 或唯一媒体路径确认入库。
5. TaskStore 已记录完整成功事件。

任意条件不满足，自动巡检只能停止在当前阶段并通知，不调用删除接口。

## Web 和 Telegram

Web `/quality` 增加自动化面板：

- 自动巡检开关、执行时间、时区、处理上限。
- 最近一次运行状态、开始/结束时间、发现/修复/跳过/失败数量。
- 下次执行时间和当前 115 风控冷却状态。
- “立即巡检”和“恢复环境默认”按钮。
- 每个受影响任务显示自动修复动作、结果和失败原因，并链接到任务详情。

Telegram `/quality` 优先展示最近一次自动巡检的异常和跳过原因。异常消息附带查看任务、重试当前阶段和暂停该任务按钮；成功修复不发送每日摘要。

## 验证和测试

新增测试覆盖：

- `02:50` 默认时间、可配置时间和 IANA 时区校验。
- 同一天重复调用只产生一次运行记录，跨日期正常执行；重启后不会重复执行。
- 两个调度线程同时触发时只有一个获得运行权。
- 自动巡检与正常任务抢占同一资源时安全跳过。
- 缺失目录、直链 STRM、异常分享 STRM 的自动修复入队结果。
- 失效分享只在明确失效时重建，风控和未知错误不重建。
- 新 STRM 或 Emby 验证失败时不会删除旧 STRM/115 源。
- 115 调用上限和冷却期间不继续扫描。
- Web 设置、运行摘要和 Telegram 异常通知内容。

验收标准：

- 运行一次模拟全流程后，TaskStore 能看到扫描、修复、跳过和失败的完整记录。
- 重复执行同一天的调度不会产生重复分享、重复 STRM 或重复删除。
- 所有自动修复路径都通过现有全量测试和 Docker 构建。
