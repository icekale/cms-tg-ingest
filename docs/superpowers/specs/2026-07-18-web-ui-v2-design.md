# Web UI V2 统一重构设计

## 背景

`cms-tg-ingest` Web UI 已具备运行概览、任务详情、本地质量巡检和 TaskStore 健康页面，但当前呈现仍有三个明显问题：

1. 首页“需要关注”和“当前队列”重复展示同一批异常任务，页面过长。
2. 任务详情把常用信息、技术字段和高风险操作放在同一层级，定位卡点和选择操作都不够直接。
3. 巡检与健康页以大段原始文本为主，数百条逐文件问题难以扫读。

本次采用已确认的 **A「状态优先控制台」** 方向，统一重构全部 Web 页面。实现继续使用 Python 服务端渲染，不引入前端框架、构建链或新的外部请求。

## 目标

- 用户在 3 秒内判断系统是否正常、任务引擎是否可用、115 是否处于风控冷却。
- 用户在 1 分钟内定位任务停在哪个阶段、为什么慢、推荐采取什么操作。
- 首页异常任务与活跃队列相互去重。
- 巡检结果从逐文件文本转为按任务聚合，同时保留完整诊断信息。
- 危险操作与普通操作明确隔离，并保留二次确认。
- 桌面和移动端均能完整使用，不隐藏关键状态或操作。

## 非目标

- 不修改任务引擎、CMS/115/Emby 工作流或 TaskStore 数据模型。
- 不新增配置中心、复杂搜索、历史分页或图表。
- 不增加自动刷新、高频轮询、115 扫描或其他外部 API 调用。
- 不引入 React、Vue、Svelte、CSS 框架、图标依赖或前端构建步骤。
- 不调整现有 URL、POST 行为、`WEB_TOKEN` 校验或部署结构。

## 共同页面框架

所有页面使用同一套服务端渲染框架：

- 顶部品牌名统一为“CMS 入库助手”。
- 固定显示三个现有入口：运行概览 `/`、质量巡检 `/quality`、本地健康 `/health`。
- 当前入口使用文字颜色和底部边线标识，不使用大面积彩色导航。
- 内容区保持单层、全宽布局；面板仅用于任务列表、摘要和诊断工具，不嵌套卡片。
- 页面背景使用近中性色，边框替代重阴影，圆角控制在 8px 以内。
- 状态色只用于成功、运行、需处理、失败和风控冷却，不形成单色主题。
- 不加载外部字体、图片、脚本或图标资源。

`_page` 增加当前导航项参数，由各渲染函数声明 active 状态。导航链接和页面标题保持服务端输出，避免引入客户端路由。

## 首页：状态优先控制台

### 顶部状态

首页标题下方展示一条系统状态带：

- 总体状态：运行正常、正在处理或需要关注。
- 简短说明：当前活跃任务数或问题任务数。
- TaskEngine 状态。
- 115 风控冷却状态；冷却时显示剩余时间。

这些信息仅来自 `list_recent_tasks`、现有任务元数据和 `build_task_health`，页面渲染不得请求 115、CMS 或 Emby。

### 指标

保留四项可扫读指标：

- 运行中
- 需处理/失败
- 等待资源
- 已完成历史

指标只反映最近 100 个 TaskStore 任务，与当前数据读取边界一致。

### 双栏规则

“需要关注”和“当前队列”继续并列，但使用互斥规则：

- **需要关注**：`FAILED`、`NEEDS_ACTION` 和不在自动调度队列中的异常活跃任务。
- **当前队列**：可调度的 `PENDING`、`RUNNING` 任务，包括资源锁、延迟检查和文件稳定等待。
- 同一任务不能同时进入两栏。

需要关注默认显示前 8 项；超过 8 项时使用原生 `<details>` 在原页面展开剩余任务，不能因为去重而让旧问题不可访问。当前队列最多显示 25 项，符合低并发使用场景。

任务行展示：标题、任务号、状态、当前阶段、简短卡点、执行/等待耗时、115 调用次数。活跃任务额外显示八阶段进度条。

### 任务阶段

UI 将内部阶段合并为八个用户可理解的阶段：

1. 接收：`RECEIVED`、`CMS_SUBMITTED`
2. CMS 整理：`ORGANIZING`、`ORGANIZED`
3. 分类识别：`RECOGNIZING`
4. 建分享：`SHARE_ALIAS_PREPARED`、`OWN_SHARE_CREATED`、`SHARE_VALIDATED`
5. 分享 STRM：`SHARE_SYNC_SUBMITTED`、`STRM_READY`、`CMS_DELETE_SETTLED`
6. 移动入库：`MOVED`
7. Emby 确认：`EMBY_CONFIRMED`
8. 清理完成：`CLEANED`

`NEEDS_ACTION` 和 `FAILED` 是结果状态，不作为第九个流程阶段。进度优先从任务事件判断最后完成阶段；信息不足时只显示当前问题状态，不虚构已完成进度。

### 本地维护

清理历史从队列标题旁移到页面底部“本地维护”区域：

- “重新载入页面”只进行普通 GET 刷新。
- “清理已结束记录”保留现有 POST 路由和确认提示。
- 文案明确说明只清除已结束任务记录，不删除媒体文件。

## 任务详情：解释优先、操作分级

### 首屏结构

任务详情首屏按以下顺序展示：

1. 面包屑、任务号、文件夹标题和状态。
2. 当前卡点提示，使用一段简短原因和下一步建议。
3. 仅在当前确实存在推荐操作时展示一个主操作。
4. 八阶段进度条。
5. 任务摘要和最近事件。

主操作继续由 `decide_retry` 和当前任务状态决定，不在 UI 中复制工作流判断。没有有效自动操作时只显示说明，不伪造可点击按钮。

### 任务摘要

首层信息只包含：

- 当前阶段
- 目标媒体库
- 为什么慢
- 执行耗时
- 115 本阶段/累计调用次数
- 推荐操作

目标路径、错误码、分阶段耗时和分阶段调用统计放入“技术详情与文件路径”原生折叠区。长路径和错误文本必须转义并允许换行，不能撑破布局。

### 操作安全

- 推荐的低风险恢复或重试操作使用唯一主按钮。
- “查 Emby”等辅助操作使用次级按钮。
- 只渲染当前状态允许的操作，不显示无效的禁用按钮。
- “从头重跑”放入独立的“高风险操作”折叠区，并继续使用浏览器确认。
- 所有现有 POST 路由、重定向和副作用保持不变。

### 时间线

最近 3 至 8 条事件默认显示，最新事件在前。更早事件放入“查看更早事件”折叠区，仍在同一页面可访问。每条事件展示阶段、状态、简短消息和时间；原始错误必须转义。

## 质量巡检：按任务聚合

`/quality` 继续调用现有 `scan_task_quality(store)`，但不再把完整报告作为首屏主体。

### 摘要

页面展示：

- 问题总数
- 受影响任务数
- 目标目录缺失数
- 直链 STRM 数
- 非预期分享 STRM 数

### 聚合列表

按照 `task_id` 聚合 `QualityIssue`：

- 每行展示任务号、标题、问题类型及各类型数量。
- “查看”链接进入现有 `/task/<id>`。
- 完整逐文件报告保留在原生折叠区，便于诊断具体路径。
- 无问题时展示明确的健康空状态，不显示空表格。

“修复全部问题”保留现有 `/quality/fix` 和确认提示。页面明确说明该巡检只读取本地 TaskStore 和 STRM 文件，不扫描 115；点击“重新巡检”只是重新加载当前 GET 页面。

## 本地健康：结构化状态

`/health` 使用现有 `build_task_health` 生成结构化摘要，并继续通过 `format_task_health` 提供完整原始报告。

首屏展示：

- TaskEngine 是否启用
- 115 是否处于风控冷却及剩余时间
- 待执行、运行中、需人工、锁等待数量
- 最近问题或最近锁等待任务

完整健康文本放入“查看完整健康报告”折叠区。页面仍然只读取本地 TaskStore，不主动请求 115、CMS 或 Emby。

## 响应式与可访问性

- 桌面宽度下首页保持“需要关注 / 当前队列”并列。
- 小于 760px 时按“需要关注 → 当前队列 → 本地维护”堆叠为单列。
- 顶部三个导航入口保持可见，可换行但不使用隐藏菜单。
- 指标在移动端使用两列网格。
- 任务详情摘要改为单列；阶段条允许横向滚动，标签不互相覆盖。
- 表格型巡检结果在移动端改为任务块，不要求横向滚动才能看到操作。
- 按钮最小高度 36px，键盘焦点清晰，正文和状态色满足基础 WCAG AA 对比度。
- 尊重 `prefers-reduced-motion`；本次不引入持续动画。
- 字号使用固定响应式断点，不随视口连续缩放；字间距保持 0。

## 数据流与性能边界

页面渲染只使用已有本地数据：

- `store.list_recent_tasks(limit=100)`
- `store.list_events(task_id)`
- `build_task_health(store, enabled=True)`
- `scan_task_quality(store)`
- 现有任务诊断、重试决策和标题辅助函数

CSS 与 HTML 继续内嵌在 Python SSR 输出中。客户端只使用原生链接、表单、`<details>` 和 `confirm()`。不增加自动轮询、WebSocket、外部静态资源或 JavaScript 状态管理。

## 错误处理

- 不存在的任务继续返回产品化的“任务不存在”页面。
- 非法任务 ID 继续保持 404 行为。
- 无任务、无异常、无巡检问题均提供清晰空状态。
- 结构化健康或巡检摘要失败时，不吞掉错误；保留现有诊断报告作为降级内容。
- 所有用户可见标题、路径、错误和事件消息继续通过 `html.escape` 输出。

## 实现范围

主要修改：

- `app/web.py`：共享页面框架、首页、任务详情、巡检、健康及响应式样式。
- `tests/test_web_admin.py`：页面结构、互斥分类、聚合巡检、健康摘要、操作可见性和安全确认。

如阶段进度或质量聚合需要纯函数辅助，优先放在 `app/web.py` 内部；只有出现明确复用时才移动到独立模块。本次不做无关后端重构。

## 验收标准

1. 首页同一任务不会同时出现在“需要关注”和“当前队列”。
2. 超过 8 个问题任务时，其余任务仍可在首页展开访问。
3. 首页能展示 TaskEngine 和 115 风控冷却状态，且渲染不触发外部请求。
4. 任务详情首屏能看到卡点、推荐动作和八阶段进度。
5. “从头重跑”和“清理已结束记录”均位于明确的危险区域并保留确认。
6. 巡检页按任务聚合问题，完整逐文件报告仍可展开查看。
7. 健康页结构化显示队列和风控状态，完整原始报告仍可展开查看。
8. 360px、768px 和 1440px 宽度下无横向页面溢出、文字遮挡或按钮挤压。
9. 现有 GET/POST 路由、Token 行为和任务副作用全部保持兼容。
10. Web 单测和完整测试集通过。

## 验证方式

```sh
python3 -m unittest tests.test_web_admin -v
python3 -W error::ResourceWarning -m unittest discover -s tests -v
```

实现后使用 Playwright 检查 `/`、典型 `/task/<id>`、`/quality` 和 `/health` 的桌面与移动端截图，并确认页面渲染期间没有新增 115、CMS 或 Emby 网络请求。
